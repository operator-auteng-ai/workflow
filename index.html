Core Principle

Everything has its place. No duplication.

Each doc owns a domain. References between docs, never copies.

VISION.md
Owns: product scope, users, pricing, success metrics
ARCH.md
Owns: system diagrams, DB schema, data flow, boundaries
PLAN.md
Owns: phased implementation order, exit criteria
TAXONOMY.md
Owns: canonical names, enums, shared vocabulary
CODING-STANDARDS.md
Owns: quality bars, security rules, definition of done
WORKFLOW.md
Owns: session protocol, read order, handoff process
🔗
PLAN.md references ARCH for what to build — never re-describes the architecture.
🚫
If you're writing something that exists in another doc, you're doing it wrong.
The Hierarchy

Start high level. Expand as you go.

1
VISION.md highest level

What are we building and why? Product scope, target users, pricing, principles.

2
ARCH.md system level

How does it work? Mermaid diagrams, component boundaries, DB schema, data flow.

3
PLAN.md execution level

In what order? Phases, dependencies, exit criteria. References ARCH — never re-describes it.

The pattern
● VISION says what to build
● ARCH says how it works
● PLAN says what order
Each level zooms in. None repeats what's above.
ARCH.md

Architecture and diagrams. That's it.

No class-level design. No implementation code. No prose that belongs elsewhere.

✓ What goes in ARCH
📊
Mermaid diagrams — component, flow, sequence level
📂
System boundaries — what talks to what
🗃️
DB schema — tables, relations, key fields
🔄
Data flow — how information moves through the system
✗ What does NOT go in ARCH
🚫
Class diagrams or method signatures
🚫
Implementation code or pseudo-code
🚫
Product decisions (that's VISION)
🚫
Implementation order (that's PLAN)
github.com/.../ARCH.md | PLAN refs this for what to build. Claude reads it and knows the system.

Iteration

MVP ships. Then iterate again.

Create docs/v1/ — same structure, next version.

project structure
saas/
├── docs/ # MVP — current arch
│ ├── ARCH.md # system of record
│ ├── PLAN.md # refs ARCH
│ ├── VISION.md # product scope
│ └── ...
│
└── docs/v1/ # NEXT version
├── v1_ARCH.md # future state diff
├── v1_PLAN.md # refs v1_ARCH + ARCH
└── v1_VISION.md # what's new in v1
How v1_ARCH.md works
  • ▶ Shows the future state of the architecture
  • ▶ References project-level ARCH — doesn't duplicate it
  • ▶ Only describes what changes — an architecture diff
  • ▶ v1_PLAN.md refs both v1_ARCH and project ARCH
